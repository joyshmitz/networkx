## Підсумок

На момент `v1` close-out intake layer уже не є лише generate-and-compile flow. На цьому milestone один workspace проходить повний human-facing loop:

- збір role-based answers;
- компіляція в canonical questionnaire і normalized reports;
- оцінка baseline readiness через `preview`;
- маршрутизація unresolved items і findings у review packets;
- оцінка evidence quality і, де policy це дозволяє, застосування narrow blocking gate;
- discoverability generated outputs через stable workspace manifest.

Це вже не planning concept, а executable operator baseline з документованою командною поверхнею і verified regression suite.

## Що Доставив `v1`

| Можливість | Основні команди й артефакти | Чому це важливо |
| --- | --- | --- |
| Регенерація workbook structure без втрати відповідей | `project/intake generate ... --preserve-responses` | дозволяє оновлювати workbook layout без втрати вже внесених значень |
| Shared workspace snapshot | `project/src/intake/workspace_snapshot.py` | дає `preview`, `review` і `evidence` один authoritative upstream data layer |
| Baseline readiness preview | `project/intake preview`, `reports/preview_status.*` | відповідає, чи готовий workspace до baseline discussion і downstream synthesis |
| Routed review packets | `project/intake review`, `reports/reviewer_registry.*`, `reports/review_packet.*` | перетворює unresolved fields, validator findings і evidence gaps на accountable work items |
| Advisory evidence status | `project/intake evidence`, `reports/evidence_status.*` | робить quality of evidence видимою до пізніх acceptance stages |
| Narrow blocking evidence gate | `project/intake evidence` exit semantics + evidence policy | дозволяє blocking лише для explicit allowlisted fields на explicit late stages |
| Generated artifact index | `reports/workspace.manifest.yaml` | робить generated outputs deterministic і discoverable без створення нового manual input file |

## Поточний Операторський Контракт

Поточна operator-facing поверхня така:

```bash
project/intake generate <workspace> [--date YYYY-MM-DD] [--preserve-responses]
project/intake compile <workspace> [--date YYYY-MM-DD]
project/intake preview <workspace> [--date YYYY-MM-DD]
project/intake review <workspace> [--date YYYY-MM-DD]
project/intake evidence <workspace> [--date YYYY-MM-DD]
project/intake verify [pytest args...]
project/intake demo happy [--date YYYY-MM-DD]
project/intake demo stress [--date YYYY-MM-DD]
```

`project/intake init` тут навмисно відсутній. Його додано вже в наступній bootstrap/CF-2 фазі, після цього close-out.

На close-out важливі три semantics:

1. `preview` показує readiness і pipeline state, але не застосовує blocking evidence semantics.
2. `evidence` є єдиною operator command, яка може завершитися non-zero через missing evidence, і це відбувається лише в narrow tested blocking scope.
3. `review` і `evidence` генерують reports, але не створюють ще один editable source of truth.

## Поточний Blocking Evidence Scope

Blocking evidence gate у `v1` навмисно лишається вузьким.

| Правило | Поточна поведінка |
| --- | --- |
| Allowlisted fields | `fat_required`, `sat_required` |
| Minimum blocking strength | `workspace_artifact` |
| Advisory-only stages | `concept`, `basic_design` |
| Blocking-allowed stages | `detailed_design`, `build_commission` |
| Surface where gate applies | лише `project/intake evidence` |

Це свідома межа. `v1` не вводить blanket evidence gate для всіх полів і всіх stages.

## Що Лишилося Поза Scope

`v1` не включає:

- notifications або chat/email delivery;
- external task systems;
- workflow bots або background automation поза repo outputs;
- blanket evidence gate для всіх полів;
- free-text evidence parsing heuristics;
- нові intake modes, не пов'язані з поточним human-facing workflow.

## Closing Statement

`v1` завершений як робочий intake layer baseline. Подальшу роботу після цього close-out слід оформлювати як нову planning задачу, а не як прихований хвіст усередині старого `v1` plan.
