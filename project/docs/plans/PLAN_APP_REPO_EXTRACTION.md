## Призначення

Цей план визначає, коли і як прикладний intake-продукт має перестати жити всередині суверенного форка `networkx` і перейти у власний репозиторій.

План навмисно починається з неприємної правди:

- продукт уже живе в `project/`;
- бібліотека `networkx` уже не є продуктом;
- але продукт досі операційно паразитує на кореневому бібліотечному repo, його editable package і його tooling.

Це працює як тимчасовий етап. Як постійна модель це погано.

Цей документ є активним лише тому, що теперішня стратегічна проблема змінилася. Bootstrap та CF-2 вже реалізовані. Планувати їх повторно не треба. Треба довести продукт до стану, у якому він перестане залежати від бібліотечного кореня як від операційного протеза.

## Поточний Стан, Перевірений У Репозиторії

### Що вже правда

- Майже всі продуктові зміни відносно `main` живуть у `project/` і пов'язаних стратегічних документах.
- В `project/src/` код імпортує `networkx` як бібліотеку через публічний Python API.
- Ми не ведемо окрему продуктову розробку всередині `networkx/`.
- `project/pyproject.toml` і `project/requirements.txt` уже декларують `networkx` як явну залежність продукту.
- Встановлювана Python-поверхня продукту вже звужена до `network_methodology_sandbox`, а legacy top-level install surface прибрана.
- Основний залишковий операційний борг уже не в Python package layout. Shell wrapper `project/intake` уже доведено до product-root-aware стану для `tests/`, `examples/` і operator-facing command hints. Окремий copied-root rehearsal зі свіжим product-local `.venv` і `pip install -e '<copied-root>[dev]'` уже підтвердив, що `verify` і `demo` можуть відпрацювати без опори на монорепо `.venv`. Але wrapper усе ще живе всередині цього repo і, поки триває monorepo-stage, зберігає parent-repo `.venv` як compatibility fallback.

### Що це означає

- Ми вже відокремили продукт концептуально і значною мірою операційно.
- Найбільший технічний шматок decoupling уже виконано.
- Розділення repo прямо зараз уже варте репетиції; головний ризик тепер не в імпортній межі, а в shell-layer debt, фінальному installer/default-runtime contract і прихованих assumptions навколо реального split workflow.

## Головне Рішення

Цільова модель така:

- intake-продукт отримує окремий repo;
- `networkx` стає зовнішньою залежністю;
- бібліотека не вбудовується в продукт як vendor tree;
- будь-які майбутні carry-патчі до `networkx`, якщо вони з'являться, живуть окремо і явно.

## Чому Не Треба Вбудовувати Бібліотеку В Продукт

Ідея "залишимо все в одному дереві і просто назвемо це продуктом" звучить дешево лише на першому кроці.

Насправді вона дає погані наслідки:

- рев'ю змішує продуктові зміни з історією чужої бібліотеки;
- контракт пакування та інструментів лишається двозначним;
- split відкладається доти, поки не стане болючим;
- будь-який sync з upstream стає політично і технічно бруднішим;
- ми починаємо поводитися так, ніби чуже дерево коду є нашою внутрішньою платформою.

Це не архітектура, а форма відкладення рішення.

## Умови Готовності Для Відділення

Відокремлення в окремий repo вважається готовим тільки тоді, коли виконані всі ці умови:

1. `project/` має власний контракт залежностей, у якому `networkx` вказаний явно.
2. Продукт збирається, запускається і тестується без залежності від кореневого editable-пакета форка.
3. Для продукту є окремий відтворюваний шлях встановлення.
4. `project/intake verify` або його прямий наступник працює в ізольованому продуктовому середовищі.
5. Уся user-facing і operator-facing документація вже посилається на продуктовий контекст, а не на бібліотечний fork як джерело істини.

## Коли Суверенний Форк Уже Треба Завершувати

Поточний форк перестає бути довгостроково прийнятною моделлю, коли одночасно видно таке:

- майже всі активні зміни живуть у `project/`;
- корінь репозиторію вже плутає людей більше, ніж допомагає;
- нам треба окреме продуктове версіонування, roadmap або issue discipline;
- залежність на `networkx` можна pin-ити версією або ref, а не тримати як власний source tree.

Коли ці умови виконані, подальше життя в одному repo стає формою боргу, а не простоти.

## Рекомендований Порядок Робіт

### Фаза 1. Розрив Залежностей

Мета: зробити продукт самодостатнім, не відділяючи репозиторій фізично.

Потрібно:

- додати явну залежність на `networkx` у продуктовий контракт залежностей;
- перестати покладатися на кореневий editable package;
- зафіксувати окремий шлях встановлення й тестування для продукту;
- перевірити, що всі продуктові тести проходять у такому режимі.

Станом на теперішній baseline ця фаза вже майже закрита: dependency contract, canonical namespace, cleanup legacy install surface і verification в чистому `cwd` уже доставлені. [PLAN_APP_DEPENDENCY_DECOUPLING.md](PLAN_APP_DEPENDENCY_DECOUPLING.md) слід читати як execution record цієї хвилі та як реєстр одного залишкового боргу навколо `project/intake` і repo-local `.venv`, а не як головний фронт нової нагальної роботи.

### Фаза 2. Продуктова Межа Пакування

Мета: зробити `project/` схожим на окремий продукт навіть до фізичного відділення.

Потрібно:

- визначити, що є коренем продукту після відділення;
- мінімізувати залежності від шляхів до кореня репозиторію;
- вирішити долю shell wrapper `project/intake` у новому repo;
- вирівняти документацію під продуктову структуру.

### Фаза 3. Репетиція Відділення

Мета: перевірити майбутнє відділення до справжнього відділення.

Потрібно:

- змоделювати новий repo layout;
- прогнати встановлення, `verify` і прикладні сценарії в репетиційній структурі;
- зафіксувати всі приховані залежності на старий root.

Перші практичні rehearsal slices цієї фази вже прибрали два найпростіші самообмани:

- shell wrapper більше не жорстко пришитий до `project/tests` і `project/examples`, а операторські підказки можуть показувати `intake ...` у extracted-like layout замість вічного `project/intake ...`;
- copied-root harness `project/tests/rehearse_extracted_install.py` уже доводить жорсткіший сценарій: temp-root copy, fresh `.venv`, `pip install -e '<copied-root>[dev]'`, після чого `verify`, `demo happy` і `demo stress` проходять без опори на монорепо `.venv`.

Це все ще не split і не остаточний installer/default-runtime contract, але це вже реальна перевірка нового layout, а не черговий документ про нього.

### Фаза 4. Реальне Відділення Репозиторію

Мета: перевести продукт у власний repo без втрати історії рішень.

Потрібно:

- створити новий repo;
- перенести продуктову частину в його корінь;
- підключити `networkx` як зовнішню залежність;
- перенести мінімально потрібні docs, tests і examples;
- залишити в цьому форку явне посилання на новий продуктовий репозиторій.

## Чого Цей План Не Робить

- не вимагає негайного repo split сьогодні;
- не маскує проблему залежностей під "ще одну функціональну фазу";
- не пропонує hard fork `networkx` без реальної технічної потреби;
- не змішує extraction work з новими intake-функціями.

## Найнебезпечніша Помилка

Найгірший сценарій не в тому, що ми split зробимо занадто рано.

Найгірший сценарій у тому, що ми ще довго будемо жити в зручній брехні:

- продукт нібито окремий;
- але збірка залежить від бібліотечного repo-root;
- метадані репозиторію вже говорять "app";
- а реальна розробка все ще прив'язана до чужої бібліотечної форми.

Це саме той вид невидимого боргу, який потім вибухає під час CI, release або onboarding.

## Критерії Завершення Поточної Фази

Поточна фаза цього плану вважається виконаною, коли:

- створено окремий план виконання для розриву залежностей;
- активні документи вже посилаються на цей план як на наступний стратегічний крок;
- історичні планувальні документи більше не видають себе за поточну дорожню карту;
- у repo більше немає двозначності, де активна продуктова вісь і де історичний бібліотечний спадок.
