## Для чого цей документ потрібен тепер

Тепер цей документ потрібен лише як історичний слід того, як ми мислили відразу після `v1` close-out.

Його не слід використовувати як current planning surface. Причина проста:

- bootstrap gap, навколо якого тут побудована частина логіки, уже закритий;
- post-`v1` planning contour уже змінився;
- активна стратегічна задача тепер не "як добудувати bootstrap", а "як відокремити продукт від бібліотечного форка".

## Для чого потрібен цей документ

Після завершення `v1` з'явилися дві різні задачі, які легко змішати, якщо їх не розвести прямо.

Перша задача, далі вона називається **A**, це жива перевірка поточного workflow. Тут ми не придумуємо нову функціональність, а дивимося, чи поточний процес справді проходиться руками на реальному або тестовому workspace без прихованого знання реалізації.

Друга задача, далі вона називається **B**, це planning наступної фази після `v1`. Тут питання вже інше: не "чи працює те, що є", а "що має бути наступною осмисленою фазою, якщо baseline уже зібраний".

Цей документ потрібен саме для цього розведення:

- **A** дає факти про те, як поводиться поточний baseline;
- **B** допомагає вирішити, що робити після baseline;
- жоден із цих блоків не повинен підміняти собою user-facing operator guide.

## Поточна база

На цій стадії repo вже має закритий `v1` baseline:

- role-based generate/compile flow;
- preview status;
- routed review packets;
- advisory evidence layer;
- narrow blocking evidence gate;
- workspace manifest;
- operator-facing documentation;
- green regression suite.

Це важливо не для святкування, а для дисципліни. Якщо не назвати базу прямо, дуже легко продовжити ставитися до `v1` так, ніби він досі "майже готовий", і непомітно перетворити нову роботу на нескінченний хвіст старого плану.

## A. Fresh Live Rehearsal

### Для чого це робити

Fresh live rehearsal потрібен, коли ми хочемо ще раз пройти поточний workflow руками і перевірити не лише correctness, а й реальну придатність до користування.

Ключове питання тут таке: чи може людина, яка працює з об'єктом і документами, пройти шлях `generate -> fill -> compile -> preview -> review -> evidence` без зайвого тертя і без внутрішньої пам'яті про те, як усе влаштовано всередині.

### Що має бути на виході

- короткий фактологічний висновок про те, чи workflow справді проходиться руками;
- список operator friction, якщо він виявився;
- окремо перевірений blocking path;
- окремо перевірений unblock path;
- розуміння, чи є знайдені проблеми maintenance-level, чи вони впливають на post-v1 planning.

### Що це не означає

Fresh live rehearsal не означає, що `v1` незавершений. Він означає лише те, що ми хочемо ще раз перевірити поточний baseline у живому проході.

## Перехід від A до B

B має починатися не з фантазій і не зі списку всього, що "було б добре колись зробити". Він має починатися з фактів, які ми отримали в A.

Якщо rehearsal пройшов добре, B можна формувати як новий planning contour поверх стабільного baseline.

Якщо rehearsal знайшов реальні workflow problems, їх треба або виправити як follow-up до baseline, або явно винести в окремий список, але не маскувати під нову "велику фазу".

## B. Post-v1 Next Planning Task

### Для чого це робити

Після close-out наступна корисна робота має бути оформлена як новий план, а не як невиразне "ще трохи допиляємо `v1`". Це потрібно і для ясності scope, і для чесного управління очікуваннями.

### Що має бути на виході

- короткий problem statement для post-v1 фази;
- чіткий список того, що входить у scope;
- окремий список того, що свідомо не входить у scope;
- один пріоритетний напрямок, а не набір рівноправних побажань;
- rough milestone breakdown для наступної фази.

### Типові кандидатні напрями

- workflow polish і зменшення operator friction;
- зовнішня доставка або notifications поза repo outputs;
- розширення evidence policy;
- інтеграції з зовнішніми task systems;
- нові intake modes або нові класи generated artifacts.

### Уже зафіксований follow-up item

Один із конкретних workflow gaps уже виявився на практиці: `project/intake generate` не bootstrap-ить новий workspace з порожнього шляху.

Поточна поведінка така:

- папка workspace повинна існувати до запуску `generate`;
- у ній уже повинен лежати `role_assignments.yaml`;
- якщо цього немає, оператор отримує помилку `Workspace not found` або не може дійти до генерації workbook-ів.

Чому це важливо:

- людина природно очікує, що `generate` є стартовою командою для нового об'єкта;
- реальна поведінка вимагає ручного bootstrap-кроку, який зараз неочевидний без читання docs або exemplar-ів;
- це створює friction саме в той момент, коли користувач уперше пробує workflow.

Поточний безпечний workaround:

- створити папку workspace вручну;
- додати `role_assignments.yaml`;
- лише потім запускати `project/intake generate`.

Напрямок майбутнього рішення треба оформлювати окремо. Найочевидніші варіанти:

- explicit `init` або `bootstrap` command;
- або розширення `generate`, щоб він умів створювати мінімальний workspace за явним шаблоном;
- або окремий generator для стартового `role_assignments.yaml`.

## Як цим документом користуватися

Якщо вам потрібно зрозуміти, як практично працювати з workspace, зупиніться тут і відкрийте `INTAKE_OPERATOR_GUIDE.md`.

Якщо вам потрібно:

- перевірити, як поточний baseline поводиться у живому проході, йдіть у блок **A**;
- вирішити, що є наступною planning-задачею після `v1`, йдіть у блок **B**.

Найпростіша формула така:

- **operator guide** відповідає на питання "що мені робити з workspace";
- **master note** відповідає на питання "як нам перевіряти baseline і як планувати після нього".
