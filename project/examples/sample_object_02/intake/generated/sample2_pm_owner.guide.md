## metadata (Phase 1)

### criticality_class
**Клас критичності** | S4

**Призначення:** Визначає базову жорсткість resilience, security і operations вимог.

**Правило вибору:** Обирається за наслідком втрати сервісу, а не за бажаністю резервування.

**Інтерпретація:** Піднімає нижню межу для redundancy, OOB, logging і acceptance evidence.

**Впливає на:** resilience, security, operations

**Значення:**

- `low` — Низька
  Коротка втрата сервісу допустима без значного впливу.
  *Обирайте якщо: Обирається лише для справді некритичних контурів.*
- `medium` — Середня
  Деградація допустима, але контрольована і коротка.
  *Обирайте якщо: Обирається для значущих, але не місійно-критичних сервісів.*
- `high` — Висока
  Втрата сервісу істотно впливає на об'єкт або операторів.
  *Обирайте якщо: Обирається для operationally critical objects.*
- `mission_critical` — Критична
  Втрата сервісу неприйнятна без деградованого або резервного режиму.
  *Обирайте якщо: Обирається для об'єктів з високою ціною відмови.*

### object_id
**Ідентифікатор об'єкта** | S4

**Призначення:** Стабільний машинний ідентифікатор об'єкта для всіх артефактів.

**Правило вибору:** Має бути унікальним в межах програми.

**Інтерпретація:** Використовується як root key для requirements, graph artifacts і downstream packs.

**Впливає на:** identity, traceability

---

### object_name
**Назва об'єкта** | S3

**Призначення:** Людська ідентифікація об'єкта в документах і звітах.

**Правило вибору:** Використовується офіційна або договірна назва.

**Інтерпретація:** Потрапляє у титули і summaries downstream artifacts.

**Впливає на:** documentation

---

### object_type
**Тип об'єкта** | S4

**Призначення:** Визначає базовий клас архітектури і коло можливих archetypes.

**Правило вибору:** Обирається один основний тип, що найкраще описує виробничу функцію об'єкта.

**Інтерпретація:** Використовується для archetype preselection і scope adjacent volumes.

**Впливає на:** archetype_selection, network_volume_scope

**Значення:**

- `generation` — Генеруючий об'єкт
  Об'єкт виробництва електроенергії або енергоносія.
  *Обирайте якщо: Обирається, якщо основна виробнича функція пов'язана з генерацією.*
- `substation` — Підстанція / розподільчий об'єкт
  Об'єкт передачі або розподілу електроенергії.
  *Обирайте якщо: Обирається, якщо ядром є комутація/розподіл.*
- `utility_process` — Допоміжний технологічний об'єкт
  Насосна, котельня, допоміжна станція або інший process utility site.
  *Обирайте якщо: Обирається для допоміжної виробничої інфраструктури.*
- `industrial_site` — Промисловий майданчик
  Об'єкт з більш загальною промисловою або складською функцією.
  *Обирайте якщо: Обирається для mixed-purpose industrial sites.*

### project_stage
**Стадія проєкту** | S3

**Призначення:** Визначає допустимий рівень невизначеності і деталізації.

**Правило вибору:** Обирається поточна фактична стадія.

**Інтерпретація:** Впливає на unknown policy і stage gates.

**Впливає на:** governance

**Значення:**

- `concept` — Концепт
  Початкове опрацювання, допускає більше невизначеності.
  *Обирайте якщо: Обирається на ранньому етапі before engineering freeze.*
- `basic_design` — Базове проєктування
  Формується baseline рішень і ключові constraints.
  *Обирайте якщо: Обирається, коли architecture вже синтезується.*
- `detailed_design` — Детальне проєктування
  Рішення мають бути достатньо точними для build packs.
  *Обирайте якщо: Обирається, коли готуються implementation packs.*
- `build_commission` — Реалізація / ПНР
  Проєкт перейшов у build, FAT, SAT і closure.
  *Обирайте якщо: Обирається під час інсталяції та вводу в експлуатацію.*

## acceptance_criteria (Phase 3)

### acceptance_evidence_class
**Клас доказів приймання** | S3

**Призначення:** Визначає, який рівень доказовості потрібен для closing stage.

**Правило вибору:** Обирається за договором і критичністю об'єкта.

**Інтерпретація:** Впливає на FAT/SAT pack і as-built closure evidence.

**Впливає на:** acceptance

**Значення:**

- `basic_checklists` — Базові чеклісти
  Достатньо checklist-level evidence.
  *Обирайте якщо: Обирається для менш критичних deliverables.*
- `test_records` — Протоколи тестів
  Потрібні формальні записи результатів перевірок.
  *Обирайте якщо: Базовий production class.*
- `witnessed_protocols` — Формальні протоколи з погодженням
  Потрібні погоджені FAT/SAT protocols.
  *Обирайте якщо: Обирається для critical contractual deliveries.*
- `regulated_evidence` — Посилена доказова база
  Потрібні жорсткіші evidence і closure rules.
  *Обирайте якщо: Обирається для regulated or mission-critical environments.*

## governance (Phase 3)

### evidence_maturity_class
**Клас зрілості доказової бази** | S2

**Призначення:** Показує, наскільки questionnaire базується на фактах, а не припущеннях.

**Правило вибору:** Обирається за найслабшою критичною частиною доказової бази.

**Інтерпретація:** Впливає на readiness for baseline freeze.

**Впливає на:** governance

**Значення:**

- `assumption_heavy` — Багато припущень
  Значна частина baseline ще ґрунтується на припущеннях.
  *Обирайте якщо: Типово для concept stage.*
- `mixed` — Змішана база
  Частина рішень підтверджена, частина ще потребує уточнення.
  *Обирайте якщо: Типово для basic design.*
- `mostly_confirmed` — Переважно підтверджено
  Критичні частини мають достатню доказову базу.
  *Обирайте якщо: Очікуваний рівень перед baseline freeze.*
- `field_verified` — Підтверджено обстеженням / реалізацією
  Дані звірені з field reality.
  *Обирайте якщо: Очікуваний стан для late design / as-built.*

### waiver_policy_class
**Клас політики винятків** | S2

**Призначення:** Визначає, як поводитися з відхиленнями від baseline rules.

**Правило вибору:** Обирається відповідно до governance model проєкту.

**Інтерпретація:** Впливає на unknown handling, stage gates і closure requirements.

**Впливає на:** governance

**Значення:**

- `strict` — Жорстка політика винятків
  Винятки рідкісні, з коротким терміном дії і compensating controls.
  *Обирайте якщо: Обирається для high/mission critical programs.*
- `controlled` — Керована політика винятків
  Винятки допустимі, але строго трасуються.
  *Обирайте якщо: Базовий production-friendly варіант.*
- `provisional` — Тимчасова політика
  На ранніх стадіях можливі тимчасові waivers до stage gate.
  *Обирайте якщо: Обирається переважно для early design stages.*

## operations (Phase 3)

### asbuilt_package_required
**Потрібен as-built package** | S3

**Призначення:** Фіксує closure obligation після монтажу і ПНР.

**Правило вибору:** Для production systems значення yes є очікуваним.

**Інтерпретація:** Активує as-built closure rules і downstream closure pack.

**Впливає на:** asbuilt_closure

**Значення:**

- `yes` — Так
  Вимога або властивість входить у scope і має бути врахована в baseline.
  *Обирайте якщо: Обирається лише за наявності реальної вимоги, а не "про запас".*
- `no` — Ні
  Вимога або властивість не входить у scope поточного об'єкта.
  *Обирайте якщо: Обирається лише якщо рішення справді не потрібне.*
- `tbd` — Не визначено
  Даних або рішення поки недостатньо.
  *Обирайте якщо: Допустимість визначається unknown policy конкретного поля.*
