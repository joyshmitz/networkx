## critical_services (Phase 1)

### control_required
**Дистанційне керування входить у scope** | S4

**Призначення:** Відрізняє control path від суто telemetry path.

**Правило вибору:** Значення yes обирається, якщо через мережу мають проходити команди керування.

**Інтерпретація:** Підвищує вимоги до zoning, latency і survivability.

**Впливає на:** service_graph, security, resilience

**Значення:**

- `yes` — Так
  Вимога або властивість входить у scope і має бути врахована в baseline.
  *Обирайте якщо: Обирається лише за наявності реальної вимоги, а не "про запас".*
- `no` — Ні
  Вимога або властивість не входить у scope поточного об'єкта.
  *Обирайте якщо: Обирається лише якщо рішення справді не потрібне.*
- `tbd` — Не визначено
  Даних або рішення поки недостатньо.
  *Обирайте якщо: Допустимість визначається unknown policy конкретного поля.*

## time_sync (Phase 2)

### sync_protocol
**Протокол синхронізації** | S4

**Призначення:** Визначає базовий timing technology choice.

**Правило вибору:** Обирається за timing requirement конкретного об'єкта, а не за загальною звичкою.

**Інтерпретація:** Визначає timing overlay, equipment fit і downstream timing dependencies.

**Впливає на:** time_sync, equipment_compatibility

**Значення:**

- `ntp` — NTP
  Базова мережна синхронізація часу.
  *Обирайте якщо: Обирається, якщо високоточний timing path не потрібен.*
- `ptp` — PTP
  Потрібна підвищена точність і чутливість до timing path.
  *Обирайте якщо: Обирається для time-sensitive applications.*
- `mixed` — Змішаний
  Частина підсистем живе на PTP, частина на NTP.
  *Обирайте якщо: Обирається, якщо вимоги різних підсистем відрізняються.*

---

### timing_accuracy_class
**Клас точності синхронізації** | S3

**Призначення:** Задає точнісну межу для time-sync design.

**Правило вибору:** Обирається найжорсткіший реальний сценарій.

**Інтерпретація:** Визначає, чи достатньо NTP, чи потрібен PTP і який equipment class допустимий.

**Впливає на:** time_sync, equipment_compatibility

**Значення:**

- `relaxed_ms` — Мілісекунди
  Достатньо узгодження на рівні мілісекунд.
  *Обирайте якщо: Підходить для базової event correlation.*
- `sub_ms` — Менше мілісекунди
  Потрібна підвищена точність нижче 1 ms.
  *Обирайте якщо: Обирається для чутливіших OT use cases.*
- `tens_of_us` — Десятки мікросекунд
  Потрібен високоточний timing path.
  *Обирайте якщо: Обирається тільки за явної вимоги.*
- `site_specific` — Спеціалізована вимога
  Потрібна окрема timing note поза базовим словником.
  *Обирайте якщо: Обирається, якщо стандартні класи не описують випадок.*
