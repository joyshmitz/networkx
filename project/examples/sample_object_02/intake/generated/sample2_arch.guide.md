## external_transport (Phase 2)

### carrier_diversity_target
**Ціль різнорідності операторів / каналів** | S4

**Призначення:** Фіксує рівень допустимої залежності від одного каналу або carrier.

**Правило вибору:** Обирається за required resilience, а не за поточним availability у полі.

**Інтерпретація:** Використовується у WAN design і failure-domain checks.

**Впливає на:** resilience, external_transport

**Значення:**

- `single_path_allowed` — Один канал допустимий
  Допускається залежність від одного зовнішнього шляху.
  *Обирайте якщо: Обирається лише для некритичних або тимчасових сценаріїв.*
- `dual_carrier_preferred` — Два канали бажані
  Другий канал бажаний, але не завжди обов'язковий.
  *Обирайте якщо: Обирається, якщо resilience потрібен, але не абсолютний.*
- `dual_carrier_required` — Два незалежні канали обов'язкові
  Один carrier/route не вважається достатнім.
  *Обирайте якщо: Обирається для critical transport paths.*
- `common_cause_separated` — Потрібна різнорідність і рознесення
  Резерв має бути рознесений не лише логічно, а й по common-cause domains.
  *Обирайте якщо: Обирається для найжорсткіших resilience targets.*

### transport_separation_policy
**Політика розділення транспортів** | S4

**Призначення:** Визначає, чи CCTV/OT/IIoT можуть ділити фізичний або логічний транспорт.

**Правило вибору:** Обирається за risk tolerance і criticality.

**Інтерпретація:** Впливає на zoning, PoE/uplink pressure і admissible paths.

**Впливає на:** logical_graph, service_graph, resilience

**Значення:**

- `shared_ok` — Спільний транспорт допустимий
  Сервіси можуть ділити транспорт за умови загальних базових правил.
  *Обирайте якщо: Обирається лише при низькому risk envelope.*
- `logical_separation` — Потрібне логічне розділення
  Спільна фізика допускається, але потрібні VLAN/VRF/ACL/QoS boundaries.
  *Обирайте якщо: Типовий варіант для mixed environments.*
- `physical_separation` — Потрібне фізичне розділення
  Критичні сервіси не повинні ділити той самий транспортний каркас.
  *Обирайте якщо: Обирається для control-first або високоризикових об'єктів.*
- `hybrid_controlled` — Частково спільний, частково розділений
  Різні сервіси мають різні вимоги до separation.
  *Обирайте якщо: Обирається лише якщо це явно погоджено.*

### wan_required
**Потрібен зовнішній транспорт** | S4

**Призначення:** Визначає, чи об'єкт потребує WAN/external connectivity.

**Правило вибору:** Значення yes обирається, якщо об'єкт не є суто локальним.

**Інтерпретація:** Активує external transport baseline і carrier-related checks.

**Впливає на:** external_transport

**Значення:**

- `yes` — Так
  Вимога або властивість входить у scope і має бути врахована в baseline.
  *Обирайте якщо: Обирається лише за наявності реальної вимоги, а не "про запас".*
- `no` — Ні
  Вимога або властивість не входить у scope поточного об'єкта.
  *Обирайте якщо: Обирається лише якщо рішення справді не потрібне.*
- `tbd` — Не визначено
  Даних або рішення поки недостатньо.
  *Обирайте якщо: Допустимість визначається unknown policy конкретного поля.*

## resilience (Phase 2)

### common_cause_separation_required
**Потрібне рознесення common-cause domains** | S3

**Призначення:** Фіксує, що резерв не повинен ділити спільні failure domains.

**Правило вибору:** Значення yes обирається для об'єктів, де паперовий резерв не приймається.

**Інтерпретація:** Активує future common-cause validation layer.

**Впливає на:** failure_domain

**Значення:**

- `yes` — Так
  Вимога або властивість входить у scope і має бути врахована в baseline.
  *Обирайте якщо: Обирається лише за наявності реальної вимоги, а не "про запас".*
- `no` — Ні
  Вимога або властивість не входить у scope поточного об'єкта.
  *Обирайте якщо: Обирається лише якщо рішення справді не потрібне.*
- `tbd` — Не визначено
  Даних або рішення поки недостатньо.
  *Обирайте якщо: Допустимість визначається unknown policy конкретного поля.*

### redundancy_target
**Ціль резервування** | S4

**Призначення:** Визначає мінімальний рівень survivability для інфраструктури.

**Правило вибору:** Обирається за tolerated failure impact, а не за бажаністю "мати резерв".

**Інтерпретація:** Породжує resilience baseline, failure-domain rules і SAT obligations.

**Впливає на:** resilience, failure_domain

**Значення:**

- `none` — Без резервування
  Допускається single-path / single-node design для некритичних випадків.
  *Обирайте якщо: Обирається лише за низької критичності.*
- `uplink_backup` — Резерв зовнішнього каналу
  Потрібно пережити втрату одного uplink/carrier.
  *Обирайте якщо: Обирається для basic resilient transport.*
- `active_node_backup` — Резерв критичного вузла
  Один вузол або один gateway не повинен валити сервіс повністю.
  *Обирайте якщо: Обирається для більш жорстких sites.*
- `n_plus_1` — N+1
  Потрібен структурований запасний елемент або шлях.
  *Обирайте якщо: Базовий industrial-resilient клас.*
- `no_spof` — Без SPOF
  Single points of failure неприйнятні в критичних шляхах.
  *Обирайте якщо: Обирається для high/mission critical designs.*
