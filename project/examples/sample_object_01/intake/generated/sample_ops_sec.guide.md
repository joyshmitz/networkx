## object_profile (Phase 1)

### staffing_model
**Модель присутності персоналу** | S3

**Призначення:** Визначає, наскільки об'єкт покладається на remote operations і OOB.

**Правило вибору:** Обирається фактична модель роботи після введення в експлуатацію.

**Інтерпретація:** Впливає на support model, OOB expectations і MTTR assumptions.

**Впливає на:** operations, management_oob

**Значення:**

- `local_ops` — Постійний локальний персонал
  На об'єкті є постійний staff для первинної реакції.
  *Обирайте якщо: Обирається, якщо люди реально присутні на зміні.*
- `remote_ops` — Переважно віддалена експлуатація
  Основна підтримка йде віддалено, local response обмежений.
  *Обирайте якщо: Обирається для unattended / lightly staffed sites.*
- `hybrid_ops` — Змішана модель
  Поєднується віддалена і локальна підтримка.
  *Обирайте якщо: Обирається, якщо обидва режими реально використовуються.*

## resilience (Phase 2)

### degraded_mode_profile
**Профіль деградованого режиму** | S3

**Призначення:** Фіксує, що саме повинно лишатися живим при часткових відмовах.

**Правило вибору:** Обирається найгірший допустимий, але прийнятний degraded mode.

**Інтерпретація:** Потрапляє у resilience model, commissioning pack і operations handoff.

**Впливає на:** resilience, operations

**Значення:**

- `best_effort` — Best-effort деградація
  Після відмови допустиме суттєве зниження функціональності.
  *Обирайте якщо: Обирається для low-criticality objects.*
- `telemetry_survives` — Має лишитися телеметрія
  Основний degraded mode повинен зберігати telemetry/monitoring.
  *Обирайте якщо: Типовий варіант для remotely operated sites.*
- `control_and_telemetry_survive` — Має лишитися керування і телеметрія
  Control path теж не повинен зникнути при одиничній відмові.
  *Обирайте якщо: Обирається для критичних operational sites.*
- `site_specific` — Спеціалізований профіль
  Потрібен окремий degraded mode note.
  *Обирайте якщо: Обирається, якщо стандартні профілі не покривають об'єкт.*

### mttr_target_class
**Клас цільового MTTR** | S3

**Призначення:** Визначає, наскільки швидко система має бути відновлювана.

**Правило вибору:** Обирається за фактичною сервісною моделлю, а не за оптимістичним очікуванням.

**Інтерпретація:** Впливає на OOB, support model, spares assumptions і commissioning tests.

**Впливає на:** operations, resilience

**Значення:**

- `same_day` — Протягом дня
  Відновлення того ж дня є прийнятним.
  *Обирайте якщо: Обирається для non-urgent field repair models.*
- `four_hours` — До 4 годин
  Відновлення має бути швидким, але не миттєвим.
  *Обирайте якщо: Обирається для important operational sites.*
- `one_hour` — До 1 години
  Вимагається дуже швидке відновлення або сильна remote recoverability.
  *Обирайте якщо: Обирається для critical sites.*
- `near_zero` — Майже без простою
  Архітектура має мінімізувати потребу у відновленні як окремій події.
  *Обирайте якщо: Обирається для найбільш жорстких cases.*

## security_access (Phase 2)

**Призначення:** Визначає нижню межу для logging baseline.

**Правило вибору:** Значення yes обирається, якщо policy або criticality вимагають збереження access/events trail.

**Інтерпретація:** Впливає на management, security і commissioning evidence.

**Впливає на:** security, operations

**Значення:**

- `yes` — Так
  Вимога або властивість входить у scope і має бути врахована в baseline.

- `no` — Ні
  Вимога або властивість не входить у scope поточного об'єкта.
  *Обирайте якщо: Обирається лише якщо рішення справді не потрібне.*
- `tbd` — Не визначено
  Даних або рішення поки недостатньо.
  *Обирайте якщо: Допустимість визначається unknown policy конкретного поля.*

### contractor_access_policy
**Політика доступу підрядників** | S3

**Призначення:** Визначає, як зовнішні сервісні ролі можуть взаємодіяти з системою.

**Правило вибору:** Обирається за risk appetite і фактичним support contract.

**Інтерпретація:** Впливає на firewall intent, logging і break-glass notes.

**Впливає на:** security, operations

**Значення:**

- `forbidden` — Підрядний доступ заборонений
  Зовнішні підрядники не повинні входити в production environment.
  *Обирайте якщо: Обирається, якщо support model fully internal.*
- `escorted_only` — Лише супроводжуваний доступ
  Доступ можливий тільки під контролем власної служби.
  *Обирайте якщо: Обирається для жорстко контрольованих об'єктів.*
- `time_bounded_remote` — Лише обмежений у часі віддалений доступ
  Доступ можливий на limited basis із logging та approval.
  *Обирайте якщо: Базовий варіант для controlled vendor support.*
- `managed_service` — Керований сервісний доступ
  Є постійний зовнішній сервіс за формалізованим контрактом.
  *Обирайте якщо: Обирається тільки за наявності formal support arrangement.*

### remote_access_profile
**Профіль віддаленого доступу** | S3

**Призначення:** Фіксує допустимий спосіб remote support.

**Правило вибору:** Обирається за support model і policy безпеки.

**Інтерпретація:** Впливає на bastion/VPN/OOB expectations.

**Впливає на:** security, management_oob

**Значення:**

- `none` — Віддалений доступ не допускається
  Remote support не входить у допустимий режим.
  *Обирайте якщо: Обирається лише якщо є постійний локальний staff і policy це дозволяє.*
- `controlled_vpn` — Контрольований VPN / bastion
  Remote access допускається лише через контрольовані точки доступу.
  *Обирайте якщо: Типовий production profile.*
- `oob_only` — Лише через OOB / break-glass
  Remote access допускається лише через окремий аварійний шлях.
  *Обирайте якщо: Обирається для жорстких security/operations моделей.*
- `hybrid_support` — Кілька контрольованих способів
  Поєднання VPN, bastion і/або OOB за правилами.
  *Обирайте якщо: Обирається, коли support model справді змішана.*

### security_zone_model
**Модель зонування** | S4

**Призначення:** Визначає базову trust architecture.

**Правило вибору:** Обирається не за красою, а за реальним risk envelope об'єкта.

**Інтерпретація:** Формує logical / zone topology і firewall intent baseline.

**Впливає на:** logical_graph, interface_graph

**Значення:**

- `flat` — Пласка
  Мінімальне зонування, підходить лише для простих low-risk sites.
  *Обирайте якщо: Обирається лише за низької критичності.*
- `segmented` — Сегментована
  Є окремі зони або сегменти для ключових підсистем.
  *Обирайте якщо: Базовий production-friendly варіант.*
- `dmz_centric` — З DMZ
  Є station DMZ або явний buffer between OT and external integrations.
  *Обирайте якщо: Обирається для sites з external access/integration.*
- `strict_isolation` — Жорстка ізоляція
  Потрібні суворі trust boundaries і мінімум спільних path.
  *Обирайте якщо: Обирається для high/mission critical sites.*

## operations (Phase 3)

### maintenance_window_model
**Модель вікон обслуговування** | S2

**Призначення:** Визначає, коли і як можна виконувати planned changes.

**Правило вибору:** Обирається реальний режим зупинок/доступності об'єкта.

**Інтерпретація:** Впливає на commissioning strategy і change control notes.

**Впливає на:** operations

**Значення:**

- `planned_only` — Лише планові вікна
  Зміни допускаються тільки у визначені вікна.
  *Обирайте якщо: Обирається для process-sensitive sites.*
- `limited_anytime` — Обмежені зміни можливі будь-коли
  Невеликі зміни дозволені поза суворими вікнами.
  *Обирайте якщо: Обирається, якщо operations це реально допускає.*
- `always_available_ops` — Гнучкий режим
  Сервісна модель дозволяє ширший change cadence.
  *Обирайте якщо: Обирається для lower-risk environments.*

### support_model
**Модель супроводу** | S3

**Призначення:** Визначає, хто і як буде підтримувати систему.

**Правило вибору:** Обирається фактична after-go-live model, не тимчасова на стадії впровадження.

**Інтерпретація:** Впливає на OOB, contractor policy, MTTR і operations handoff content.

**Впливає на:** operations

**Значення:**

- `inhouse` — Власна служба
  Основна підтримка виконується internal team.
  *Обирайте якщо: Обирається за фактичної експлуатаційної моделі.*
- `integrator` — Інтегратор
  Основний support дає інтегратор.
  *Обирайте якщо: Обирається за відповідного contract model.*
- `vendor_managed` — Вендорський супровід
  Критична залежність від вендора або сервісної мережі вендора.
  *Обирайте якщо: Обирається тільки якщо це реально зафіксовано.*
- `hybrid` — Змішаний супровід
  Support розподілений між кількома сторонами.
  *Обирайте якщо: Типовий варіант для більшості production systems.*
